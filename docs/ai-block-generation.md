# AI Generation для блоков CHATPLUS

Этот документ описывает, как правильно генерировать тексты для блоков через ИИ, чтобы контент был совместим с шаблонами, ownership-контуром и CMS/data-контрактом.

Главная идея:

- ИИ должен получать не только тему страницы
- ИИ должен получать **контракт блока**
- ИИ не должен “догадываться”, как выглядит блок, по общему описанию страницы

То есть генерация строится не от фразы “сделай текст для карточек”, а от набора параметров:

- `template_kind`
- `block_type`
- цель блока
- позиция блока на странице
- ограничения по длине
- стиль
- допустимые поля
- запреты

## 1. Почему это важно

Если давать модели только общую тему страницы, она часто:

- пишет слишком длинный текст
- дублирует hero
- пишет общий маркетинговый шум
- не понимает, что блок маленький
- нарушает тональность
- делает карточки неоднородными

Если давать контракт блока, результат становится намного стабильнее.

## 2. Что обязательно передавать модели

Для любого блока нужно передавать минимум:

- `template_kind`
- `block_type`
- `page_context`
- `block_position`
- `block_goal`
- `fields`
- `constraints`
- `tone`
- `do_not`

Дополнительно желательно передавать:

- `audience`
- `industry`
- `channel`
- `integration`
- `offer`
- `good_examples`
- `bad_examples`

## 3. Универсальный prompt contract

Рекомендуемый универсальный JSON-шаблон:

```json
{
  "template_kind": "pricing",
  "block_type": "proof_card",
  "page_context": "Chat Plus для омниканальных продаж",
  "block_position": "после hero, до тарифной секции",
  "block_goal": "показать прозрачность тарифа и снять страх скрытых доплат",
  "audience": "руководитель продаж или операционный менеджер",
  "fields": ["title", "text"],
  "constraints": {
    "cards_count": 3,
    "title_max_words": 5,
    "text_max_chars": 120
  },
  "tone": "деловой, ясный, без маркетинговой воды",
  "do_not": [
    "не обещать неподтвержденные результаты",
    "не писать общие фразы",
    "не дублировать hero"
  ]
}
```

## 4. Что означают поля контракта

### `template_kind`

Показывает, для какого шаблона генерируется блок:

- `home`
- `structured`
- `directory`
- `pricing`
- `partnership`
- `tenders`
- `resource-hub`
- `brand-content`
- `comparison`
- `campaign`

### `block_type`

Тип блока. Его нужно называть так, чтобы было понятно, в какой section contract он попадает.

Примеры:

- `proof_card`
- `hero_panel_item`
- `pricing_tier`
- `faq_item`
- `comparison_advantage`
- `cta_block`
- `step_card`
- `feature_card`

### `page_context`

Контекст страницы:

- какой продукт
- для кого страница
- какой оффер
- о какой интеграции/отрасли/канале идет речь

### `block_position`

Где блок находится относительно других секций:

- сразу после hero
- перед FAQ
- после pricing tiers
- в правой панели hero

Это помогает модели не дублировать соседний контент.

### `block_goal`

Что пользователь должен понять после этого блока.

Примеры:

- снять страх скрытых доплат
- быстро показать три ключевые выгоды
- объяснить, почему интеграция важна
- показать отличие от конкурента

### `fields`

Какие поля модель должна заполнить.

Примеры:

- `title`
- `text`
- `label`
- `subtitle`
- `bullet_points`

### `constraints`

Ограничения по количеству и длине.

Примеры:

- `cards_count`
- `title_max_words`
- `text_max_chars`
- `bullets_count`
- `bullet_max_chars`

### `tone`

Фиксирует стиль текста.

Обычно для CHATPLUS подходит:

- деловой
- конкретный
- уверенный
- без пустых обещаний

### `do_not`

Очень важное поле. Помогает не получить мусор.

Примеры запретов:

- не дублировать hero
- не использовать абстрактные обещания
- не писать “лучший / уникальный / революционный”
- не обещать неподтвержденные цифры
- не делать текст длиннее контракта

## 5. Рекомендуемый формат задания модели

Лучше всего передавать модели:

1. краткий контекст страницы
2. block contract
3. ограничения
4. формат ответа

Пример текстового промпта:

```text
Сгенерируй контент для блока страницы CHATPLUS.

Контекст страницы:
- продукт: Chat Plus
- сценарий: омниканальные продажи
- аудитория: руководитель продаж

Контракт блока:
- template_kind: pricing
- block_type: proof_card
- позиция: после hero
- цель: снять страх скрытых доплат и показать прозрачность тарифа
- поля: title, text
- карточек: 3
- title: до 5 слов
- text: до 120 символов
- стиль: деловой, конкретный, без воды

Нельзя:
- дублировать hero
- писать общие фразы
- обещать неподтвержденные результаты

Верни JSON-массив объектов формата:
[{ "title": "...", "text": "..." }]
```

## 6. Примеры по типам блоков

### Пример: `pricing -> proof_card`

Задача:

- показать прозрачность
- коротко снять сомнения

Хорошо:

- короткий заголовок
- одна ясная мысль
- без перегруза

Плохо:

- длинные абзацы
- лозунги
- пересказ hero

### Пример: `structured -> step_card`

Задача:

- объяснить этап внедрения или пользовательский flow

Лучше просить:

- короткий заголовок шага
- одно практическое объяснение

### Пример: `comparison -> advantage`

Задача:

- показать отличие от конкурента

Важно:

- опираться на факты страницы
- не выдумывать неподтвержденные claims
- не скатываться в агрессивный маркетинг

### Пример: `resource-hub -> CTA block`

Задача:

- перевести пользователя к следующему действию

Важно:

- CTA не должен повторять весь hero
- текст должен быть коротким и направляющим

## 7. Что стоит хранить как block generation metadata

Идеально для каждого блока иметь не только сам контент, но и генерационный контракт:

- `block_type`
- `content_goal`
- `field_schema`
- `length_rules`
- `tone_rules`
- `good_examples`
- `bad_examples`

Это можно хранить:

- в документации
- в prompt-templates
- в generator-layer

Не обязательно прямо в Strapi schema, если редактору это поле не нужно.

## 8. Что нельзя делать

- не генерировать контент без понимания `block_type`
- не давать модели только “тему страницы” без ограничений
- не использовать ИИ-текст как source of truth без проверки на длину и смысл
- не подменять контракт блока свободным длинным копирайтингом
- не вставлять сгенерированный текст в шаблон напрямую, если блок CMS-owned

## 9. Как проверять результат генерации

Перед записью в seeds или Strapi нужно проверить:

1. текст реально помещается в блок
2. нет дублирования hero/CTA/FAQ
3. тональность соответствует шаблону
4. количество карточек/пунктов совпадает с контрактом
5. нет запрещенных claims и общей воды
6. JSON соответствует ожидаемым полям

## 10. Практическое правило для проекта

В CHATPLUS правильный способ генерации такой:

- сначала определить block contract
- потом сгенерировать контент по contract-aware prompt
- потом проверить длину и смысл
- потом только записывать в source seed или managed CMS-поле

Если модель не знает:

- зачем блок нужен
- где блок стоит
- сколько текста влезает
- что нельзя писать

она почти наверняка даст результат хуже, чем можно получить при том же самом контексте страницы.
