# Карта интерфейса Strapi для ручной работы

Этот документ отвечает на вопрос: "Куда нажимать в Strapi, если нужно вручную изменить или добавить страницу?"

Сейчас основной режим работы - ручной. AI-генерацию не используем как обычный редакторский процесс, пока она отдельно не включена и не проверена.

Рабочие адреса:

- публичный сайт: `https://astro.integromat.ru`
- Strapi admin: `https://strapi.integromat.ru/admin`

Если вы редактор, вам не нужно запускать команды. Всё, что нужно для контента, находится внутри Strapi.

## Главное меню слева

### Dashboard

Стартовый экран после входа.

Можно использовать, чтобы:

- быстро открыть последние измененные страницы;
- убедиться, что вы вошли под правильным пользователем.

Обычно редактировать сайт с Dashboard не нужно.

### Content Manager

Главный рабочий раздел.

Здесь редактируются:

- страницы сайта;
- блоки внутри страниц;
- сущности каталога;
- blueprints;
- версии страниц.

Если вы редактор, почти вся работа идет через `Content Manager`.

### Media Library

Хранилище картинок и файлов.

Используйте для:

- OG-картинок;
- изображений в hero;
- иллюстраций;
- файлов, которые нужно прикрепить к странице.

### Settings

Технические настройки Strapi.

Редактору обычно не нужно сюда заходить. Исключения:

- сменить пароль;
- проверить профиль;
- администратору создать API token;
- администратору настроить роли и права.

## Где редактировать страницы

Путь:

```text
Content Manager -> Page
```

Что там делать:

- найти страницу по `route_path`;
- изменить текст;
- добавить или удалить блок;
- поменять порядок блоков;
- настроить SEO;
- включить или выключить страницу в меню;
- включить или выключить страницу в sitemap;
- опубликовать или снять публикацию.

Примеры поиска:

| Что ищем | Что ввести в поиск |
| --- | --- |
| Главная | `/` |
| Цены | `/pricing` |
| Партнерам | `/partnership` |
| Сравнение Respond.io | `/compare/respond-io` |
| Канал Email | `/channels/email` |

## Где создавать новую страницу

Путь:

```text
Content Manager -> Page -> Create new entry
```

Порядок:

1. Заполните `title`.
2. Заполните `slug`.
3. Заполните `route_path`.
4. Выберите `page_kind`.
5. Выберите `blueprint`.
6. Заполните SEO.
7. Добавьте блоки в `sections`.
8. Настройте меню и sitemap.
9. Сохраните.
10. Проверьте страницу.
11. Поставьте `editorial_status = approved`.
12. Нажмите `Publish`.

Подробные примеры лежат в [Рецептах заполнения страниц](strapi-page-recipes.md).

## Где редактировать блоки страницы

Путь:

```text
Content Manager -> Page -> открыть страницу -> sections
```

`sections` - это список блоков сверху вниз. Как блоки идут в Strapi, так они идут на странице.

Что можно делать:

- раскрыть блок и изменить текст;
- добавить новый блок через `Add component`;
- перетащить блок выше или ниже;
- удалить блок, если он больше не нужен;
- сохранить страницу.

Важное правило: на старых страницах не удаляйте блоки вслепую. Сначала проверьте страницу в браузере, потому что старый URL может иметь уникальный макет.

## Где редактировать меню

Меню управляется полями внутри самой `Page`.

Путь:

```text
Content Manager -> Page -> открыть страницу -> nav/show fields
```

Поля:

| Поле | Что делает |
| --- | --- |
| `show_in_header` | Показывает страницу в верхнем меню |
| `show_in_footer` | Показывает страницу в футере |
| `nav_group` | Выбирает группу меню |
| `nav_label` | Текст ссылки |
| `nav_description` | Короткое описание для расширенного меню или футера |
| `nav_order` | Порядок внутри группы |

Если страница не должна быть в меню, выключите `show_in_header` и `show_in_footer`.

## Где редактировать sitemap

Путь:

```text
Content Manager -> Page -> открыть страницу -> show_in_sitemap
```

Правила:

- для обычной публичной страницы обычно `show_in_sitemap = true`;
- для тестовой страницы обычно `false`;
- для страницы с `robots = noindex` обычно `false`;
- для дубля или служебного route обычно `false`.

## Где редактировать SEO

Путь:

```text
Content Manager -> Page -> открыть страницу -> SEO fields
```

Поля:

| Поле | Что писать |
| --- | --- |
| `seo_title` | Короткий заголовок для поиска |
| `seo_description` | Описание примерно 120-160 символов |
| `canonical` | Основной URL, если есть дубли |
| `robots` | `index,follow` или `noindex,nofollow` |
| `og_image` | Картинка для соцсетей |
| `hreflang_policy` | Языковые версии, если нужны |

Минимум перед публикацией: `seo_title` и `seo_description`.

## Где редактировать каталоговые сущности

Путь:

```text
Content Manager -> Channel / Industry / Integration / Solution / Feature / Business Type / Competitor
```

Эти сущности хранят факты.

Примеры:

- `Channel` - WhatsApp, Telegram, Email.
- `Industry` - недвижимость, медицина, образование.
- `Integration` - AmoCRM, Bitrix24.
- `Solution` - продажи, поддержка, автоматизация.
- `Feature` - AI Calendar, омниканальность.
- `Competitor` - Intercom, Respond.io, Zendesk.

Важно: сущность не заменяет страницу. Публичная страница всё равно должна жить в `Page`.

## Где смотреть blueprints

Путь:

```text
Content Manager -> Page Blueprint
```

Blueprint задает:

- тип страницы;
- допустимые блоки;
- обязательные блоки;
- стартовый набор секций.

Редактору обычно не нужно менять blueprints. Их меняют, когда команда решила добавить новый тип страницы или изменить правила конструктора.

## Где смотреть версии страниц

Путь:

```text
Content Manager -> Page Version
```

`Page Version` хранит snapshot страницы.

Используется для:

- истории изменений;
- диагностики;
- rollback;
- понимания, что было опубликовано раньше.

Обычному редактору обычно не нужно создавать версии вручную. Они создаются системой.

## Что делать после изменения страницы

1. Нажмите `Save`.
2. Проверьте страницу в браузере.
3. Убедитесь, что нет заглушек и пустых блоков.
4. Поставьте `editorial_status = approved`.
5. Нажмите `Publish`.
6. Дождитесь rebuild/deploy, если работает production flow.

## Что делать, если изменения не видны на сайте

Проверьте:

- нажали ли `Save`;
- нажали ли `Publish`;
- стоит ли `editorial_status = approved`;
- не выключена ли страница через `migration_ready`;
- не открыта ли старая cached версия;
- прошел ли rebuild/deploy.

Для старых страниц дополнительно:

- `migration_ready` должен быть включен только после проверки;
- `parity_status` должен быть `approved`;
- если эти поля не включены, сайт может показывать legacy fallback.

## Что нельзя менять без понимания

Не меняйте случайно:

- `route_path` у старой страницы;
- `legacy_template_family`;
- `legacy_layout_signature`;
- `migration_ready`;
- `parity_status`;
- `source_mode`;
- системные blueprints;
- snapshots в `Page Version`.

Если сомневаетесь, лучше изменить текст внутри блока, но не менять системные поля.

## Быстрая шпаргалка

| Задача | Куда идти |
| --- | --- |
| Изменить текст на странице | `Content Manager -> Page -> sections` |
| Добавить блок | `Content Manager -> Page -> sections -> Add component` |
| Создать страницу | `Content Manager -> Page -> Create new entry` |
| Убрать страницу | `Content Manager -> Page -> Unpublish` |
| Добавить в меню | `Content Manager -> Page -> show_in_header/show_in_footer/nav_*` |
| Изменить SEO | `Content Manager -> Page -> seo_*` |
| Добавить картинку | `Media Library` |
| Посмотреть историю | `Content Manager -> Page Version` |
| Посмотреть правила блоков | `Content Manager -> Page Blueprint` |

## Что с AI

AI-генерация будет отдельным будущим процессом. В ручной работе сейчас не нужно создавать `Generation Job`.

Когда AI включится, он должен будет создавать обычные `Page`-черновики, которые проверяются по тем же правилам: URL, SEO, блоки, ссылки, меню, sitemap и безопасность старых макетов.
