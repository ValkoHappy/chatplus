# CHATPLUS для владельца

Это самый короткий нетехнический вход в проект.

Используйте его, если вы:

- владелец проекта
- менеджер
- партнер
- контент-оператор без желания читать полную инженерную документацию

## 1. Что это за проект

`CHATPLUS` — это публичный сайт продукта `Chat Plus`.

У него есть две части:

- публичный сайт, который видят посетители
- CMS-админка `Strapi`, где меняется контент

Простая схема:

- `site` = страницы для пользователей
- `CMS` = место, где команда меняет контент

## 2. Как теперь устроен контент

В проекте есть три режима записей:

- `managed` — редактируется руками в `Strapi`
- `imported` — загружается системой, но хранится в `Strapi`
- `settings` — системные singleton-настройки

Главный смысл:

- редактор работает в `Strapi`
- сайт собирается из `Strapi`
- importer помогает массово загружать SEO/catalog данные
- importer не должен слепо уничтожать ручные правки

## 3. Что обычно меняют руками

Вручную в CMS обычно меняют:

- landing pages
- singleton pages
- site settings

Catalog/SEO сущности вроде `competitor`, `solution`, `channel` и похожие:

- тоже живут в `Strapi`
- но для них действует safe-sync модель
- это не “свободный ручной каталог”, а управляемый импортируемый слой

## 4. Как выглядит обычное обновление

### Если кто-то изменил контент в Strapi

Нормальный flow:

1. изменить запись в `Strapi`
2. нажать `Publish`
3. webhook идет в relay
4. CI пересобирает сайт
5. сайт обновляется

То есть целевая модель уже не про “кто-то помнит правильную последовательность команд”, а про предсказуемый publish flow.

### Если кто-то изменил import/source данные

Тогда оператор:

1. обновляет source data
2. смотрит importer plan
3. запускает importer
4. публикует изменения
5. rebuild выкатывает сайт

## 5. Что уже автоматизировано

В проекте уже есть:

- воспроизводимый Docker/VPS deploy
- importer с режимами `plan / apply / force-sync / report`
- publish relay
- CI workflows для content publish и code publish
- backup/restore scripts
- production upload provider под S3-compatible storage

## 6. Что пока все еще контролирует человек

Человек все еще нужен для:

- принятия решения, что именно публиковать
- visual QA
- настройки ролей, токенов и webhook при первом запуске
- управления сложными импортами и `force-sync`

Это нормально: система уже зрелая, но еще не полностью “на автопилоте”.

## 8. Самое короткое резюме

- сайт статический
- контентом управляет `Strapi`
- `Strapi` — главный редакторский интерфейс
- importer нужен для массовых SEO/catalog обновлений
- публикация должна идти через `Publish -> webhook -> rebuild -> deploy`
