# Матрица маршрутов и ownership

Используйте этот файл как быстрый reference по ownership публичных routes и статусу migration.

## Правила ownership

- `managed` означает, что запись редактируется вручную в `Strapi`
- `imported` означает, что запись создаётся importer-ом и затем поддерживается в `Strapi` по sync-правилам
- `settings` означает singleton и system records

Frontend не должен становиться вторым источником истины для page copy.

## Bridge-правило для legacy managed-маршрутов

Для migratable managed routes runtime resolution теперь работает так:

- если на exact route есть published `page_v2`, рендерится `page_v2`
- иначе рендерится legacy source

Draft не может перехватить legacy route.

## Политика защиты от поломок

- migration идёт через bridge, а не через destructive rewrite
- legacy wrappers остаются compatibility layer на весь этап
- один route = одно publish decision
- route считается migrated только после parity smoke
- rollback делается снятием published state у `page_v2`, без emergency code rollback

## Что нельзя делать

- не создавайте imported catalog entities вручную как основной workflow
- не добавляйте новые manual routes по привычке через legacy `landing-page`
- не меняйте ownership между `managed` и `imported` без одновременного обновления docs и validation
- не считайте legacy navigation metadata главным источником истины после миграции route в published `page_v2`

## Когда обновлять этот файл

Обновляйте матрицу, если:

- появляется новый публичный route
- route меняет ownership
- route входит или выходит из bridge migration
- вводится новая immutable reserved family
- route family переходит из legacy ownership в page-first ownership
